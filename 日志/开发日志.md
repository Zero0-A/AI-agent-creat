#### 决策 1：选择「单 Agent（对话流模式）」
- **理由**：严格按编排执行、保留多轮历史、适合结构化任务
- **排除**：自主规划模式（不稳定）、多 Agents（过度设计）

- 默认变量：
  - `USER_INPUT` (String) → ✅ 勾选必填，承载用户本轮输入
  - `CONVERSATION_NAME` (String) → 不使用，可保留或删除

#### 步骤 4：完整性检查节点（代码节点）配置中
- **方案**：B（拆 8 个字段输入，而非整 Object）
- **输入**：8 个变量（major / grade / skills / available_time / award_goal / preferred_direction / project_experience / existing_assets），分别通过 ⊕ 引用画像提取节点对应字段
- **代码**：`async def main(args)` 检查 5 个必填字段（major / grade / skills / available_time / award_goal）
- **输出**：
  - `is_complete` (Boolean)
  - `missing_fields` (Array\<String\>)
  - `missing_count` (Integer)
  - `profile` (Object 透传)
- **判定规则**：None / 空字符串 / 空数组 都视为缺失；`preferred_direction` 非必填
- **节点试运行 V1**：输出结构正确，但 profile 全空（节点单独试运行未填 mock 数据，需端到端试运行验证）
- **端到端运行 V1**：profile 仍全空，missing_fields 包含所有 5 个必填字段
- **根因定位**：输入区把 8 个字段嵌套在 `profile1` 父对象下（截图显示 profile1 → major/grade/...），但代码用 `params.get('major')` 在根层级找，导致取不到值
- **修复方案**：采用方案 2 - 删除 profile1 父级，将 8 个字段重建为顶层平级变量，每个通过 ⊕ 单独引用画像提取节点对应字段；代码保持原版
- **端到端运行 V2**（输入"我是计算机专业大二，会 Python 和 C++，想冲省赛三等奖"）：
  - `is_complete: false`
  - `missing_fields: ["available_time"]`
  - `missing_count: 1`
  - `profile` 8 字段全部正确填充
- **状态**：✅ 完整性检查节点定稿（v1）

#### 步骤 5.5：升级为方案 C（必填硬检查 + 选填软提醒分级）
- **决策**：维持 5 必填 + 3 选填的字段配置；3 选填不阻塞流程，但作为软提醒在推荐生成后引导补充
- **必填字段**：major / grade / skills / available_time / award_goal（缺失则阻塞 → 进入追问）
- **选填字段**：preferred_direction / project_experience / existing_assets（缺失不阻塞，仅作软提醒）
- **代码升级**：新增 is_empty 辅助函数，分别检查 required_fields 和 optional_fields
- **输出新增 3 个字段**：
  - `soft_missing_fields` (Array\<String\>)：选填缺失列表
  - `soft_missing_count` (Integer)：选填缺失数量
  - `has_soft_gap` (Boolean)：是否有选填缺失
- **下游影响**：
  - 选择器节点：仍只看 `is_complete`，软提醒不影响主分支
  - 推荐生成节点：当 `has_soft_gap=true` 时，在推荐末尾追加"💡 进一步精准推荐"引导（基于 soft_missing_fields 动态生成）
- **设计理念**：用户先看到推荐成果（承诺一致性），再被引导补充选填，转化率高于前置追问
- **状态**：⏳ 代码与输出字段升级中

#### 步骤 6：选择器节点（条件分支）— 开发中
- **节点类型**：选择器 / 条件 / IF
- **作用**：根据 `is_complete` 分流
- **输入**：`is_complete` (Boolean) → 引用 完整性检查节点.is_complete
- **分支规则**：
  - 分支 1：`is_complete == true` → 知识库检索（节点 6，后续开发）
  - 分支 2（ELSE）：`is_complete == false` → 追问生成（节点 5）
- **当前阶段连线**：分支 1 暂连临时结束节点；分支 2 连追问生成
- **开发策略**：优先打通追问路径（选择器 → 追问生成 → 结束），再回来开发推荐路径
- **状态**：✅ 选择器节点定稿

#### 步骤 7：追问生成节点（大模型节点）— 开发中
- **位置**：选择器 false 分支下游
- **节点类型**：大模型（豆包·2.0·lite，Temperature=0.5）
- **作用**：当必填字段缺失时，友好追问用户
- **输入**：
  - `profile` (Object) → 引用 完整性检查.profile
  - `missing_fields` (Array\<String\>) → 引用 完整性检查.missing_fields
  - `USER_INPUT` (String) → 引用 开始节点.USER_INPUT（用户原话上下文）
- **系统提示词 v2（用户优化版，已同步到提示词体系 4.2）**：
  - 加入字段含义对照表（每个字段附 2-4 个示例值，提升提取准确性）
  - 引入 `{{USER_INPUT}}` 提供原话上下文
  - 问题前缀改为 1️⃣2️⃣ 数字 emoji（更易读，提高视觉吸引力）
  - 强制复述已知信息（建立信任感）
  - 每个问题带 2-3 个示例选项（降低用户认知负担）
  - 新增 3 个 few-shot 示例（覆盖缺 2 项 / 缺 1 项 / 缺 5 项全部场景）
  - 硬规则 8：严禁追问 3 个选填字段
  - 硬规则 9：必须为 missing_fields 中每个字段生成对应问题，不能漏问
- **严禁追问**：preferred_direction / project_experience / existing_assets（选填字段留待推荐后软提醒收集）
- **输出**：`question` (String) — 追问文本
- **状态**：✅ 配置中（待 3 组 case 测试验证：缺 1 项 / 缺 2 项 / 缺 5 项全缺）
