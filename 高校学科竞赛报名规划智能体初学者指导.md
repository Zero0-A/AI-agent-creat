# 高校学科竞赛报名规划智能体初学者指导文档

> 本文档基于 `2026_AIC.pdf` 中“高校学科竞赛报名规划智能体原型开发”任务整理，面向智能体领域初学者，目标是帮助零基础学习者理解任务、规划学习路径，并逐步完成一个可演示、可测试、可交付的智能体原型。

***

## 0. 你最终要做出什么？

你需要开发一个面向大学生的“高校学科竞赛报名规划智能体”。

它不是普通聊天机器人，而是一个能完成以下任务的智能体：

1. 询问并理解用户情况；
2. 提取用户画像，例如专业、年级、技能、时间、目标奖项等；
3. 查询你整理好的竞赛知识库；
4. 根据用户情况推荐合适的竞赛和赛道；
5. 说明推荐理由、风险和优先级；
6. 生成阶段性备赛计划；
7. 如果用户信息不完整，主动追问；
8. 至少包含一个真实代码或工具节点，不能完全依赖提示词聊天。

最终你需要交付：

- 智能体原型链接或本地演示方式；
- README 使用说明；
- 工作流截图或架构图；
- 知识库整理文档；
- 主要提示词文档；
- 代码或工具节点说明；
- 至少 20 条测试记录；
- 一份智能体生成的推荐报告样例；
- 3-5 页汇报 PPT。

***

## 1. 智能体基本概念与文档任务解读

### 1.1 什么是智能体？

对初学者来说，可以先把智能体理解为：

> 一个能根据用户目标，自动完成“理解问题、调用知识、执行步骤、生成结果、检查结果”的 AI 应用。

普通聊天机器人通常只负责“回答问题”。

智能体更强调：

| 能力   | 说明                | 本项目中的体现                |
| ---- | ----------------- | ---------------------- |
| 目标理解 | 理解用户真正想完成什么       | 用户想参加比赛、冲奖、保研加分或项目落地   |
| 信息提取 | 从自然语言中提取结构化字段     | 提取专业、年级、技能、时间、项目经历等    |
| 知识检索 | 从知识库中查找可靠资料       | 查询竞赛规则、报名时间、赛道要求等      |
| 决策推荐 | 根据规则和用户情况做判断      | 推荐比赛、赛道、优先级            |
| 工具调用 | 使用代码或工具完成计算、解析、校验 | 匹配评分函数、JSON 提取、CSV 解析等 |
| 多轮交互 | 信息不足时继续追问         | 用户只说“推荐比赛”时，不直接乱推荐     |
| 结果自检 | 检查输出是否完整、是否基于知识库  | 检查是否包含推荐理由、风险、计划等      |

### 1.2 这个任务真正考察什么？

文档中明确说明，本题不是考察算法刷题，而是考察智能体开发综合能力。

你需要展示以下能力：

1. **需求分析能力**：能不能读懂“竞赛报名规划”场景；
2. **资料检索能力**：能不能自己查找竞赛资料；
3. **知识库搭建能力**：能不能把资料整理成结构化表格或文档；
4. **工作流设计能力**：能不能把智能体拆成多个处理节点；
5. **提示词工程能力**：能不能让输出稳定、清晰、少幻觉；
6. **基础工程能力**：能不能写至少一个代码或工具节点；
7. **测试迭代能力**：能不能设计测试题并不断优化；
8. **汇报表达能力**：能不能把方案讲清楚。

### 1.3 任务核心不是“做得很复杂”，而是“做得完整”

初学者容易犯的错误是：一上来就想做复杂系统。

本任务更重要的是完成一个闭环：

```text
用户输入
  ↓
用户画像提取
  ↓
信息完整性检查
  ↓
知识库检索
  ↓
竞赛匹配评分
  ↓
推荐结果生成
  ↓
备赛计划生成
  ↓
结果自检
  ↓
最终输出
```

只要这个闭环清晰、可演示、可测试，就比一个功能很多但逻辑混乱的系统更有价值。

## 2. 所需技术栈与前置知识学习清单

### 2.1 推荐技术路线

对初学者，建议优先选择“低代码平台 + 少量代码节点”的方式。

推荐方案：

| 路线                 | 适合人群       | 优点                | 风险          |
| ------------------ | ---------- | ----------------- | ----------- |
| Coze / 扣子          | 零基础、想快速做原型 | 上手快，有工作流和知识库能力    | 复杂逻辑可能受平台限制 |
| Dify               | 有一点技术基础    | 知识库、工作流、API 能力较完整 | 本地部署或配置略复杂  |
| FastGPT            | 想做知识库问答    | 知识库配置方便           | 工作流表达能力需学习  |
| LangChain + Python | 有编程基础      | 自由度高、工程感强         | 学习成本高       |
| 自写 Web 应用          | 有前后端基础     | 可控性最强             | 对初学者难度较高    |

**建议初学者选择：Coze / 扣子 或 Dify。**

如果你希望更稳妥地完成任务，可以采用：

> Coze / Dify 搭建主体智能体 + Python 编写竞赛匹配评分函数 + CSV/Excel 维护知识库。

### 2.2 必学基础知识清单

#### A. 智能体与大模型基础

需要理解：

- 什么是大语言模型；
- 什么是提示词；
- 什么是智能体；
- 什么是工具调用；
- 什么是工作流；
- 什么是 RAG / 知识库检索；
- 什么是结构化输出。

建议学习目标：

- 能写出一个清晰的系统提示词；
- 能让模型按 JSON 或 Markdown 格式输出；
- 能解释“为什么不能只做普通聊天机器人”。

#### B. 提示词工程

需要掌握：

- 角色设定；
- 任务目标；
- 输入字段说明；
- 输出格式约束；
- 示例输入输出；
- 禁止胡编乱造；
- 信息不完整时主动追问。

一个基础提示词结构可以是：

```text
你是高校学科竞赛报名规划顾问。
你的任务是根据用户画像和竞赛知识库，为大学生推荐合适竞赛、赛道和备赛计划。
如果用户信息不足，不要直接推荐，必须先追问。
回答必须包含：用户画像摘要、推荐竞赛、推荐理由、风险分析、准备周期、优先级、备赛计划。
所有竞赛信息必须基于知识库，不确定时说明“不确定，需要进一步查证”。
```

#### C. 知识库整理

需要掌握：

- 如何查找公开竞赛资料；
- 如何提炼关键信息；
- 如何制作表格；
- 如何注明资料来源；
- 如何把资料导入智能体平台。

建议用表格维护知识库：

| 字段   | 示例                |
| ---- | ----------------- |
| 竞赛名称 | 中国国际大学生创新大赛       |
| 适合专业 | 计算机、经管、设计、电子信息等   |
| 赛道方向 | 高教主赛道、青年红色筑梦之旅等   |
| 报名时间 | 以官方通知为准           |
| 提交材料 | 商业计划书、PPT、项目原型等   |
| 适合人群 | 有项目基础、创新创业方向学生    |
| 能力要求 | PPT、答辩、商业计划书、项目落地 |
| 准备周期 | 1-3 个月或更长         |
| 冲奖难度 | 高                 |
| 资料来源 | 官方网站或通知链接         |

#### D. Python 基础

至少需要会：

- 变量、字典、列表；
- 读取 CSV / Excel；
- 写简单函数；
- 根据条件打分；
- 输出 JSON。

本项目不要求高深算法，重点是写出可解释的规则逻辑。

#### E. 测试与记录

需要掌握：

- 如何设计不同类型的用户问题；
- 如何判断智能体回答是否合格；
- 如何记录问题；
- 如何根据测试结果改提示词、知识库或代码。

## 3. 项目问题系统拆解

### 3.1 这个项目可以拆成 7 个核心问题

| 编号 | 核心问题      | 对应任务         |
| -- | --------- | ------------ |
| P1 | 用户是谁？     | 用户画像识别       |
| P2 | 用户信息够不够？  | 信息完整性检查与多轮追问 |
| P3 | 有哪些可推荐竞赛？ | 知识库构建与检索     |
| P4 | 哪些竞赛最适合？  | 匹配评分与推荐排序    |
| P5 | 为什么推荐？    | 推荐理由与风险分析    |
| P6 | 怎么准备？     | 阶段性备赛计划生成    |
| P7 | 输出是否可靠？   | 结果自检与测试迭代    |

### 3.2 用户画像字段拆解

智能体需要识别以下字段：

| 字段     | 必要性 | 示例                     |
| ------ | --- | ---------------------- |
| 专业     | 必填  | 计算机、电子信息、机械、设计、经管      |
| 年级     | 必填  | 大一、大二、大三、研一            |
| 技能基础   | 必填  | Python、算法、硬件、PPT、文案、答辩 |
| 项目经历   | 重要  | 课程设计、科研项目、创业项目、原型系统    |
| 可投入时间  | 必填  | 每周 5 小时、暑假集中准备、1 个月内   |
| 目标奖项   | 必填  | 校奖、省奖、国奖、保研加分          |
| 偏好方向   | 可选  | 算法、智能体、硬件、创新创业、设计      |

| 落地诉求   | 可选  | 是否重视产业落地、商业化、展示效果      |

| 能力要求     | 编程、算法、PPT、答辩、硬件、商业计划书等 |
| 报名时间     | 大致时间或官方通知时间            |
| 提交时间     | 作品、材料或答辩时间             |
| 作品形式     | 代码、论文、PPT、商业计划书、硬件、视频等 |
| 组队要求     | 个人或团队，人数限制             |
| 准备周期     | 短期、中期、长期               |
| 冲奖难度     | 低、中、高                  |
| 适合已有项目转化 | 是/否/部分适合               |
| 风险点      | 时间紧、竞争强、材料复杂、硬件成本高等    |
| 资料来源     | 官方链接或公开资料说明            |

## 4. 任务拆解与优先级排序

### 4.1 MVP 最小可行版本

初学者不要一开始追求完美，先完成 MVP。

MVP 需要包含：

1. 10-15 条竞赛知识库数据；
2. 用户画像提取；
3. 信息缺失时主动追问；
4. 根据规则推荐 2-3 个比赛；
5. 输出推荐理由、风险、周期、优先级；
6. 生成备赛计划；
7. 一个代码节点，例如匹配评分函数；
8. 20 条测试记录。

### 4.2 优先级排序

| 优先级 | 任务          | 原因           |
| --- | ----------- | ------------ |
| P0  | 读懂题目并确定技术路线 | 决定后续实现方式     |
| P0  | 整理竞赛知识库     | 没有知识库就无法可靠推荐 |
| P0  | 设计用户画像字段    | 决定智能体是否能理解用户 |
| P0  | 设计工作流       | 评分中占比较高      |
| P0  | 实现至少一个代码节点  | 题目硬性要求       |
| P1  | 编写提示词       | 影响输出稳定性      |
| P1  | 设计测试集       | 影响自测与迭代评分    |
| P1  | 生成推荐报告样例    | 交付要求之一       |
| P2  | 做简单前端或更美观演示 | 加分项，不是最先做    |
| P2  | 优化 PPT 美观度  | 最后集中整理       |

### 4.3 推荐开发顺序

建议按以下顺序推进：

```text
第 1 步：读题并画出工作流
第 2 步：收集并整理竞赛资料
第 3 步：设计用户画像字段
第 4 步：写提示词
第 5 步：实现匹配评分代码节点
第 6 步：搭建智能体工作流
第 7 步：准备 20 条测试问题
第 8 步：测试、记录、优化
第 9 步：整理 README、知识库文档、提示词文档
第 10 步：制作 PPT 和演示材料
```

## 5. 分阶段学习路径

### 阶段一：理解任务与智能体基础

目标：知道自己要做什么。

学习内容：

- 智能体与聊天机器人的区别；
- 工作流节点是什么；
- 知识库问答是什么；
- 用户画像是什么；
- 为什么要多轮追问。

阶段产出：

- 一张项目流程图；
- 一份任务拆解清单；
- 技术路线选择说明。

### 阶段二：资料收集与知识库搭建

目标：让智能体有可靠信息来源。

你需要收集至少 10-15 个竞赛信息，例如：

- 中国国际大学生创新大赛；
- 挑战杯；
- 蓝桥杯；
- 中国大学生计算机设计大赛；
- 全国大学生电子设计竞赛；
- 全国大学生数学建模竞赛；
- RoboMaster；
- 互联网+、创新创业类比赛；
- 学校或省级学科竞赛。

注意：具体信息应以官方公开资料为准，不能凭空编造。

阶段产出：

- 竞赛知识库表格；
- 资料来源说明；
- 适合专业、能力要求、准备周期等结构化字段。

### 阶段三：画像提取与追问逻辑

目标：让智能体先了解用户，而不是直接推荐。

需要实现：

- 从用户输入中提取画像字段；
- 判断哪些字段缺失；
- 根据缺失字段生成追问；
- 信息足够后再推荐。

阶段产出：

- 用户画像 JSON 格式；
- 信息完整性检查规则；
- 多轮追问提示词。

### 阶段四：推荐逻辑与代码节点

目标：让推荐结果有依据。

建议实现一个竞赛匹配评分函数。

评分维度可以包括：

| 维度     | 分值建议   |
| ------ | ------ |
| 专业匹配   | 0-25 分 |
| 技能匹配   | 0-25 分 |
| 时间匹配   | 0-15 分 |
| 项目基础匹配 | 0-15 分 |
| 目标奖项匹配 | 0-10 分 |
| 偏好方向匹配 | 0-10 分 |

总分 100 分。

阶段产出：

- 匹配评分代码；
- 推荐排序结果；
- 推荐理由模板。

### 阶段五：备赛计划生成

目标：让用户知道下一步怎么做。

备赛计划至少包括：

1. 阶段划分；
2. 每阶段任务；
3. 时间安排；
4. 需要完成的材料；
5. 团队分工建议；
6. 风险提醒；
7. 冲奖策略。

阶段产出：

- 备赛计划生成提示词；
- 一份完整推荐报告样例。

### 阶段六：测试、迭代与交付

目标：证明智能体能稳定工作。

需要完成：

- 至少 20 条测试问题；
- 每条记录用户问题、智能体回答、是否满意、存在问题、优化方式；
- 根据测试结果优化提示词、知识库、代码节点或工作流。

阶段产出：

- 测试记录表；
- 优化说明；
- README；
- PPT。

## 6. 分步骤实施指南

### 6.1 环境搭建

#### 方案 A：使用 Coze / 扣子

适合零基础。

步骤：

1. 注册并登录 Coze / 扣子平台；
2. 创建一个新的智能体；
3. 设置智能体名称，例如“高校竞赛报名规划助手”；
4. 编写系统提示词；
5. 创建知识库并上传竞赛资料；
6. 创建工作流；
7. 添加代码节点或插件节点；
8. 测试并发布演示链接。

#### 方案 B：使用 Dify

适合有一点技术基础的同学。

步骤：

1. 注册 Dify 云服务或本地部署 Dify；
2. 创建 Chatflow / Workflow 应用；
3. 添加开始节点、LLM 节点、知识检索节点、代码节点；
4. 导入竞赛知识库；
5. 设置输出模板；
6. 调试工作流；
7. 发布 Web App 或 API。

#### 方案 C：Python 本地原型

适合会一点编程的同学。

你可以使用：

- Python；
- pandas 读取竞赛表格；
- OpenAI API 或其他大模型 API；
- Streamlit 做简单网页；
- JSON 存储用户画像。

这种方式工程感更强，但对初学者要求更高。

### 6.2 知识库搭建步骤

#### 第一步：确定竞赛范围

建议先选择 10-15 个常见竞赛，不要一开始收集太多。

优先选择：

- 计算机类；
- 创新创业类；
- 电子硬件类；
- 数学建模类；
- 设计与文案类；
- 综合类。

#### 第二步：查找公开资料

资料来源优先级：

1. 官方网站；
2. 官方通知文件；
3. 学校教务处 / 创新创业学院通知；
4. 比赛章程；
5. 往届获奖作品介绍；
6. 可靠经验分享文章。

#### 第三步：整理成结构化表格

建议使用 CSV 或 Excel。

表头示例：

### 6.4 匹配评分代码节点

下面是一个适合初学者的简化评分函数示例。

```python

def score_competition(user, competition):
    score = 0
    reasons = []

    # 专业匹配
    if user["major"] in competition["suitable_majors"]:
        score += 25
        reasons.append("专业方向匹配")

    # 技能匹配
    matched_skills = set(user["skills"]) & set(competition["required_skills"])
    if matched_skills:
        skill_score = min(len(matched_skills) * 8, 25)
        score += skill_score
        reasons.append(f"技能匹配：{', '.join(matched_skills)}")

    # 时间匹配
    if user["available_time_level"] >= competition["time_requirement_level"]:
        score += 15
        reasons.append("可投入时间基本满足要求")
    else:
        reasons.append("可投入时间可能不足")

    # 项目基础匹配
    if user.get("has_project") and competition.get("suitable_for_project_transfer"):
        score += 15
        reasons.append("已有项目适合转化参赛")

    # 目标奖项匹配
    if user["award_goal"] in competition["suitable_award_goals"]:
        score += 10
        reasons.append("目标奖项匹配")

    # 方向偏好匹配
    if user["preferred_direction"] in competition["directions"]:
        score += 10
        reasons.append("偏好方向匹配")

    return {
        "competition_name": competition["name"],
        "score": score,
        "reasons": reasons
    }
```

初学者可以先不用追求算法复杂，只要规则清楚、可解释即可。

## 8. 常见问题与解决方案

### 问题 1：智能体总是直接推荐，不追问

原因：提示词没有明确要求“信息不足必须追问”。

解决方案：

在系统提示词中加入硬性规则：

```text
如果专业、年级、技能基础、可投入时间、目标奖项中任意 2 项以上缺失，禁止直接推荐比赛，必须先提出 3-6 个澄清问题。
```

### 问题 2：智能体会胡编报名时间

原因：模型没有被约束只能基于知识库回答。

解决方案：

- 知识库中保留资料来源；
- 对不确定时间统一输出“以官方最新通知为准”；
- 提示词中加入“不得编造报名时间”。

### 问题 3：推荐理由太空泛

原因：没有建立匹配评分规则。

解决方案：

要求输出推荐理由时必须对应具体字段：

- 专业是否匹配；
- 技能是否匹配；
- 时间是否充足；
- 是否有项目基础；
- 是否符合目标奖项；
- 是否符合偏好方向。

### 问题 4：输出格式不稳定

原因：没有固定输出模板。

解决方案：

- 使用 Markdown 模板；
- 明确每个标题必须出现；
- 加一个“结果自检节点”；
- 测试时记录格式错误并优化提示词。

### 问题 5：知识库检索不到相关内容

原因：资料切分不合理或字段太混乱。

解决方案：

- 每个竞赛单独整理成一条；
- 字段名保持统一；
- 加入关键词，例如“算法、硬件、创新创业、PPT、答辩”；
- 不要上传无结构的大段网页复制文本。

### 问题 6：代码节点不会写

解决方案：

先写最简单的匹配评分函数即可。

最低要求：

- 输入用户画像；
- 输入竞赛列表；
- 按专业、技能、时间、目标打分；
- 输出排序后的竞赛名称和分数。

不需要一开始写复杂算法。

### 问题 7：20 条测试不知道怎么设计

解决方案：

按照文档中的 12 类覆盖类型设计，每类 1-2 条即可。

重点不是问题多复杂，而是能覆盖：

- 信息完整；
- 信息缺失；
- 冲国奖；
- 时间不足；
- 能力单一；
- 有无项目；
- 不合理要求；
- 知识库无答案；
- 多轮追问。

## 14. 最后建议

对零基础学习者来说，这个任务最重要的不是“炫技”，而是完成一个清晰、可靠、可解释的智能体闭环。

请始终记住三条原则：

1. **先追问，再推荐**：信息不足时不要乱推荐；
2. **基于知识库，不胡编**：竞赛信息必须有来源；
3. **推荐要可解释**：每个推荐都要说明依据、风险和准备路径。

完成这个项目后，你不仅会做一个竞赛规划智能体，也会掌握智能体项目中非常通用的一套方法：

```text
需求分析 → 知识库整理 → 工作流设计 → 提示词编写 → 工具节点开发 → 测试迭代 → 汇报交付
```

这套方法可以迁移到学习规划、就业指导、科研助手、产品客服、企业知识库问答等很多智能体场景中。
