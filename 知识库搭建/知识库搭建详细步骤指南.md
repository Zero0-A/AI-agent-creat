## 1. 知识库搭建整体流程

```text
第 1 步：确定竞赛范围 → 选择 15-20 个常见学科竞赛
        ↓
第 2 步：查找官方资料 → 从官网 / 教务处 / 通知文件获取
        ↓
第 3 步：设计表格结构 → 确定字段名称和字段类型
        ↓
第 4 步：逐条填充数据 → 每行一个竞赛，逐列填写
        ↓
第 5 步：内容交叉验证 → 每条数据至少两个来源确认
        ↓
第 6 步：导入智能体平台 → 上传到 Coze / Dify 表格知识库
        ↓
第 7 步：测试检索效果 → 用典型问题验证召回准确性
        ↓
第 8 步：迭代优化 → 根据测试结果补充和修正数据
```

## 2. 资料搜集范围与信息来源

### 2.1 竞赛范围选择

选择竞赛时应覆盖以下类别，确保知识库能处理不同类型的用户需求：

| 类别 | 推荐竞赛示例 | 覆盖理由 |
|---|---|---|
| 计算机 / 编程类 | 蓝桥杯、ACM-ICPC、CCPC | 编程基础型用户 |
| 创新创业类 | 中国国际大学生创新大赛（原互联网+）、挑战杯 | 项目转化型用户 |
| 电子 / 硬件类 | 全国大学生电子设计竞赛、RoboMaster | 硬件特长型用户 |
| 数学建模类 | 全国大学生数学建模竞赛、美国大学生数学建模竞赛 | 建模特长型用户 |
| 设计 / 文案类 | 中国大学生计算机设计大赛、全国大学生广告艺术大赛 | 设计/PPT特长型用户 |
| 综合 / 交叉类 | 中国大学生服务外包创新创业大赛、外研社·国才杯 | 跨专业型用户 |

**最低数量要求**：至少 10-15 个竞赛。

**数量与质量的关系**：20 个精确的资料 > 50 个模糊的资料。宁可少，但要准。

### 2.2 信息来源优先级（从高到低）

| 优先级 | 来源类型 | 示例 | 可信度 |
|---|---|---|---|
| P0 | **官方网站** | competition.china.com、dasai.cn、官方公众号 | ★★★★★ |
| P1 | **官方通知文件** | 教务处 PDF、大赛章程 | ★★★★★ |
| P2 | **学校通知** | 教务处/创新创业学院公告 | ★★★★ |
| P3 | **往届获奖公示** | 官方公布的历史获奖名单 | ★★★★ |
| P4 | **权威媒体报道** | 教育部报道、主流媒体报道 | ★★★ |
| P5 | **经验分享** | 往届参赛者经验帖 | ★★（仅作参考） |

### 2.3 资料搜索方法

**搜索关键词模板：**

```text
[竞赛名称] + 章程 / 通知 / 官方 / 报名 / 2025 / 2026
```

举例：

- `蓝桥杯 大赛章程 2025`
- `挑战杯 全国大学生课外学术科技作品竞赛 通知 2026`
- `中国国际大学生创新大赛 赛道设置 参赛要求`

**搜索渠道建议：**

1. 百度搜索 + 带上 `官网` / `官方通知` 关键词
2. 学校教务处网站 → 搜索 `竞赛` / `学科竞赛` / `创新创业`
3. 中国高等教育学会官网 → 竞赛排行榜
4. 知乎 / 小红书 → `限制为只看标题中有竞赛名称的结果`
5. 竞赛聚合平台 → `赛氪`、`大学生竞赛网` 等（注意交叉验证）

### 2.4 需要搜集的信息清单

对每个竞赛，尽可能搜集以下信息：

- 竞赛全称与简称
- 主办单位
- 竞赛级别（国家级/省部级/校级）
- 赛道 / 组别设置
- 适合专业范围
- 能力要求
- 报名时间
- 作品提交时间
- 作品形式与格式要求
- 参赛形式（个人/团队、团队人数限制）
- 评审标准
- 奖项设置与获奖率
- 往届获奖作品参考
- 参赛费用
- 是否有赛前培训

## 3. 资料筛选标准

### 3.1 纳入标准

一条竞赛资料**必须同时满足**以下条件才能纳入知识库：

1. **来源可靠**：信息来自官方网站、正式通知文件或权威渠道
2. **信息可验证**：同一信息至少能在两个来源中找到佐证
3. **时效性**：为 2024-2026 年的赛制信息
4. **描述客观**：内容为事实性描述，而非主观评价
5. **字段完整**：至少覆盖 80% 的必填字段

### 3.2 排除标准

符合以下任意一条的资料应**排除**：

1. **来源不明**：无法追溯到原始出处
2. **主观推测**：个人对获奖率的猜测、对比赛难度的主观评价
3. **严重过期**：2019 年以前的赛制资料
4. **广告性质**：明显是培训机构/商业机构的推广内容

### 3.3 信息分级

| 级别 | 定义 | 示例 | 在知识库中的处理 |
|---|---|---|---|
| **确定信息** | 官方明确写明 | 参赛形式、赛道名称 | 直接记录，无需额外标注 |
| **惯例信息** | 多年一致但未在当年通知中写明 | 报名费、获奖率 | 记录并标注"据往年情况" |
| **不确定信息** | 官方未明确、或每年变化 | 具体报名日期 | 记录并标注"以官方通知为准" |
| **推测信息** | 无依据的推测 | "这个比赛比较容易获奖" | **不纳入**知识库 |

## 4. 信息分类方法

### 4.1 竞赛分类维度

每个竞赛可以按以下维度分类（即知识库中可作为筛选条件的标签）：

| 分类维度 | 可能的值 | 用途 |
|---|---|---|
| 竞赛大类 | 计算机类、数学类、电子类、创新创业类、设计类、外语类、综合类 | 快速筛选大类 |
| 竞赛级别 | 国家级 A 类、国家级 B 类、省部级、校级 | 判断含金量 |
| 能力要求类型 | 编程、算法、硬件、PPT、答辩、文案、商业计划、建模 | 匹配用户技能 |
| 作品形式 | 代码、论文、硬件实物、商业计划书、PPT、视频、研究报告 | 判断准备方向 |
| 准备周期 | 短期（<1 月）、中期（1-3 月）、长期（>3 月） | 匹配时间条件 |
| 适合项目类型 | 已有项目转化、从零开始、理论创新、工程实践 | 匹配项目基础 |
| 赛道方向 | 算法方向、硬件方向、创新创业方向、设计方向、交叉方向 | 匹配偏好方向 |

### 4.2 信息字段分类

| 类别 | 字段 | 用途 |
|---|---|---|
| **识别信息** | 竞赛名称、竞赛简称、主办单位 | 唯一标识竞赛 |
| **匹配信息** | 适合专业、能力要求、赛道方向、准备周期、冲奖难度 | 智能体匹配评分 |
| **说明信息** | 竞赛级别、参赛形式、作品形式、报名时间、提交时间 | 生成推荐说明 |
| **决策信息** | 风险点、适合项目转化、适合保研加分、费用 | 生成风险分析 |
| **来源信息** | 资料来源链接、数据更新日期 | 验证和追溯 |

## 5. 表格结构设计

### 5.1 完整版表格设计（推荐）

| 字段名 | 数据类型 | 必填 | 说明 | 示例 |
|---|---|---|---|---|
| `competition_name` | 文本 | 是 | 竞赛全称 | 蓝桥杯全国软件和信息技术专业人才大赛 |
| `competition_short_name` | 文本 | 否 | 常用简称 | 蓝桥杯 |
| `competition_level` | 文本 | 是 | 竞赛级别 | 国家级 A 类 |
| `category` | 文本 | 是 | 竞赛大类 | 计算机类 |
| `organizer` | 文本 | 否 | 主办单位 | 工业和信息化部人才交流中心 |
| `suitable_majors` | 文本 | 是 | 适合专业，逗号分隔 | 计算机科学与技术,软件工程,电子信息 |
| `required_skills` | 文本 | 是 | 能力要求，逗号分隔 | Python,Java,C/C++,算法,数据结构 |
| `tracks` | 文本 | 是 | 赛道/组别，逗号分隔 | 软件赛,电子赛,物联网赛,设计赛 |
| `entry_format` | 文本 | 是 | 作品形式，逗号分隔 | 源代码,设计文档,说明文档 |
| `team_type` | 文本 | 是 | 参赛形式 | 个人或团队（最多3人） |
| `registration_time` | 文本 | 是 | 报名时间，不确定时标注 | 每年10月-12月（以官方通知为准） |
| `submission_time` | 文本 | 是 | 提交/比赛时间 | 次年3月-4月 |
| `preparation_cycle` | 文本 | 是 | 准备周期 | 1-3个月 |
| `time_requirement_level` | 整数 | 是 | 时间要求等级 1/2/3 | 2 |
| `difficulty_level` | 文本 | 是 | 难度 | 中 |
| `award_setting` | 文本 | 否 | 奖项设置 | 省赛：一、二、三等奖；国赛：一、二、三等奖 |
| `suitable_award_goals` | 文本 | 是 | 适合的奖项目标，逗号分隔 | 省奖,国奖 |
| `suitable_for_project_transfer` | 文本 | 是 | 适合项目转化 | 是 |
| `suitable_for_baoyan` | 文本 | 否 | 对保研是否有帮助 | 是 |
| `judging_criteria` | 文本 | 否 | 评审标准重点 | 创新性40%、技术实现30%、应用价值30% |
| `risk_points` | 文本 | 是 | 主要风险点 | 编程基础要求高、备赛周期长 |
| `required_materials` | 文本 | 否 | 需要准备的材料清单 | 源代码、设计文档、作品说明视频 |
| `typical_award_rate` | 文本 | 否 | 获奖率 | 省赛约60%、国赛约15% |
| `funding_requirement` | 文本 | 否 | 参赛费用参考 | 报名费约300元 |
| `source_link` | 文本 | 是 | 主要资料来源 | https://dasai.cn/xxx |
| `data_updated_date` | 日期 | 是 | 数据更新日期 | 2025-06-15 |
| `notes` | 文本 | 否 | 备注 | 跨专业友好，可依托课程设计参赛 |

| 能力要求 | 是 | 逗号分隔 |
| 赛道方向 | 是 | 逗号分隔 |
| 报名时间 | 是 | 标注"以官方通知为准" |
| 提交时间 | 是 | |
| 准备周期 | 是 | 短期/中期/长期 |
| 难度 | 是 | 高/中/低 |
| 风险点 | 是 | |
| 适合已有项目转化 | 是 | 是/否/部分 |

### 5.3 字段填写规范

**多值字段**：使用**英文逗号**分隔，不要在值内再包含逗号。

```
正确：计算机科学与技术,软件工程,电子信息
错误：计算机科学与技术、软件工程、电子信息
```

**时间字段**：不确定时一律加后缀 `（以官方通知为准）`。

```
正确：每年 10 月 - 12 月（以官方通知为准）
错误：10 月 - 12 月
```

**难度字段**：用固定值 `高` / `中` / `低`。

```
正确：中
错误：中等难度、一般、还行
```

**时间等级字段**（用于代码节点评分）：用数字 1/2/3。

| 值 | 含义 | 对应描述 |
|---|---|---|
| 1 | 短期 | 可 1-2 周内完成，适合时间紧张的备赛 |
| 2 | 中期 | 需要 1-3 个月系统准备 |
| 3 | 长期 | 需要 3 个月以上长期投入 |

## 6. 内容填充规范

### 6.1 填充顺序

建议按以下顺序逐列填充，避免遗漏：

1. 先填**识别信息**（竞赛名称、级别、主办单位）
2. 再填**匹配信息**（适合专业、能力要求、赛道、准备周期、难度）
3. 再填**说明信息**（报名时间、提交时间、参赛形式、作品形式）
4. 再填**决策信息**（风险点、适合项目转化、适合保研加分）
5. 最后填**来源信息**（资料来源链接、更新日期）

### 6.2 填写原则

| 原则 | 说明 |
|---|---|
| **客观准确** | 只记录事实，不添加主观评价 |
| **来源可溯** | 每条重要信息都应能追溯到来源 |
| **格式统一** | 同一字段的格式在整个表格中保持一致 |
| **避免歧义** | 使用清晰明确的语言，避免模糊表述 |
| **标注不确定性** | 不确定的信息必须标注"以官方通知为准"等说明 |

### 6.3 填写示例

以"蓝桥杯"为例，一条完整的记录：

```text
竞赛名称：蓝桥杯全国软件和信息技术专业人才大赛
竞赛级别：国家级
适合专业：计算机科学与技术,软件工程,电子信息工程,自动化,物联网工程
能力要求：Python,Java,C/C++,算法,数据结构
赛道方向：软件赛,电子赛,物联网赛,设计赛
报名时间：每年10月-12月（以官方通知为准）
提交时间：次年3月-4月（以官方通知为准）
参赛形式：个人或团队，最多3人
作品形式：源代码,设计文档,说明文档
准备周期：1-3个月
时间等级：2
难度：中
奖项设置：省赛一等奖约10%、省赛二等奖约20%、省赛三等奖约30%
适合奖项目标：省奖,国奖
适合项目转化：否
适合保研加分：是
评审标准：代码质量30%、算法设计30%、创新性20%、文档完整性20%
风险点：编程基础要求高,算法能力要求高,赛道选择需针对个人特长
需准备材料：源代码,设计文档,算法说明
费用：约300元
资料来源：https://dasai.cn/（官方指定平台）
数据更新日期：2025-03-15
备注：适合编程基础扎实的学生，可在课程设计基础上备赛
```

## 7. 内容验证流程

### 7.1 单条数据验证清单

对每条竞赛数据，独立检查以下项目：

- [ ] 竞赛名称与官方全称一致
- [ ] 所有字段已填写，无空白必填项
- [ ] 多值字段使用英文逗号分隔
- [ ] 时间字段已标注"以官方通知为准"
- [ ] 难度字段使用固定值（高/中/低）
- [ ] 时间等级字段使用数字（1/2/3）
- [ ] 资料来源链接有效，可访问
- [ ] 数据来源属于 P0-P2 可信来源
- [ ] 无主观推测内容
- [ ] 无与已知权威来源矛盾的内容

### 7.2 交叉验证方法

**重要信息至少应有两个来源确认。** 方法如下：

| 信息类型 | 验证方法 |
|---|---|
| 竞赛级别 | 官网 + 学校竞赛目录对比确认 |
| 报名/提交时间 | 官网 + 至少一份学校通知双重确认 |
| 赛道设置 | 官网大赛章程 + 官网赛道说明 |
| 参赛形式 | 官网章程 + 往届参赛须知 |
| 能力要求 | 官网赛题说明 + 往届获奖者分享 |
| 奖项设置 | 官网奖项说明 + 获奖公示名单 |

### 7.3 整体知识库验证测试

完成所有数据填充后，运行以下测试：

**测试 1：知识库检索测试**

| 测试问题 | 期望能检索到的竞赛 |
|---|---|
| 计算机专业适合什么比赛？ | 蓝桥杯、ACM、计算机设计大赛、数学建模等 |
| 有现成项目可以参加什么比赛？ | 中国国际大学生创新大赛、挑战杯、服务外包大赛等 |
| 短期时间能准备什么比赛？ | 蓝桥杯（个人赛）、计算机设计大赛（有作品基础上） |
| 对 PPT 能力要求高的比赛有哪些？ | 中国国际大学生创新大赛、挑战杯、电子商务三创赛等 |

**测试 2：匹配评分逻辑测试**

- 选择一个典型用户画像
- 用代码节点的评分函数对知识库中的竞赛评分
- 检查 Top 3 推荐结果是否合理
- 检查评分依据是否可解释

**测试 3：边界情况测试**

- 冷门专业（如体育教育）是否返回低分或无匹配
- 不合理请求（如一周内冲国奖）是否有合理风险提示
- 用户非大学身份（如高中生）是否有恰当提示

### 7.4 问题修正流程

```text
发现问题
  ↓
定位原因
  知识库数据有误 → 修正数据 → 重新导入
  字段缺失 → 补充数据
  格式不规范 → 统一格式
  评分逻辑有误 → 修改代码节点
  ↓
重新测试
  ↓
记录修正内容
```

## 8. 数据维护与更新

### 8.1 更新周期

| 更新类型 | 触发条件 | 操作 |
|---|---|---|
| 常规更新 | 每学期开学前 | 确认所有竞赛的报名时间是否已公布 |
| 紧急更新 | 收到官方赛制变更通知 | 立即修改对应竞赛字段 |
| 新增竞赛 | 发现用户高频查询但知识库中没有的比赛 | 按填写规范新增一行 |
| 数据清理 | 每半年 | 删除严重过期的数据，补充新的替代竞赛 |

## 9. 常见问题与注意事项

### Q1：官方来源找不到怎么办？

- 优先使用学校教务处下发的竞赛目录清单（通常已由学校整理过）
- 如果某字段确实找不到官方信息，该字段填写"信息暂缺"而非猜测
- 对不确定的信息，在来源字段备注"待官方更新确认"

### Q2：不同来源信息冲突怎么办？

| 冲突类型 | 处理方式 |
|---|---|
| 官网 vs 学校通知 | 以官网为准，标注学校通知中的差异 |
| 以往年 vs 今年 | 以今年最新通知为准 |
| A 学校 vs B 学校 | 优先采纳直属机构的规则，标注适用范围 |
| 官网 vs 第三方 | 以官网为准 |

### Q3：竞赛信息每年变化大怎么办？

- 在备注中标注"赛制可能调整，请以当年通知为准"
- 对于报名时间等易变字段，持续关注官方通知
- 优先记录相对稳定的信息（赛道设置、参赛形式、能力要求）

### Q4：我应该收录多少竞赛才足够？

- **最低要求**：10-15 个，覆盖 4 个以上大类
- **推荐数量**：20-30 个，覆盖 6 个以上大类
- 质量永远优先于数量：20 个数据准确的竞赛远好于 40 个数据模糊的竞赛
